# 问.app 打包与图标设计方案

## 1. 目标

在现有 Swift Package 版 Claude Lite Mac App 基础上，补齐一套本机可直接使用的 macOS 应用包装层，使项目可以产出一个能在 Finder 中双击启动的 `.app`。

本次目标不是发布版，不包含签名、公证、安装器或 DMG。

本次交付的核心结果为：

- `dist/问.app`

该应用应能在当前机器上像普通 Mac 软件一样被双击启动，并显示自定义图标。

## 2. 已确认选择

### 应用名称

- 最终名称：`问.app`

### 图标方向

- 采用 `A. Centered and Calm`

### 图标视觉要求

- 米白色背景
- 圆角方形底
- 中间单个汉字：`问`
- 风格简洁、安静、克制
- 不加入复杂纹理、装饰性图形或强烈渐变

## 3. 范围

### 本次范围内

1. 为现有程序增加标准 macOS `.app` 包装输出
2. 生成固定输出目录 `dist/`
3. 在 `dist/` 中产出 `问.app`
4. 为应用补齐 `Info.plist`
5. 为应用补齐图标资源
6. 保证 Finder 中图标可见
7. 保证双击时应用可启动
8. 增加可重复执行的打包命令或脚本
9. 完成实际打包与本机验证

### 本次范围外

1. Developer ID 签名
2. notarization 公证
3. `.dmg` 安装包
4. 自动更新
5. Launch Services 深度注册优化
6. 沙盒与发布渠道适配

## 4. 设计原则

### 保持现有主程序结构稳定

当前项目已经具备：

- Swift Package 工程入口
- 核心库层
- SwiftUI 可执行程序
- 测试与 smoke 验证

因此本次不应重写主程序架构，也不应因为打包需求强行迁移到完整 Xcode 工程。

### 包装层独立

`.app` 打包逻辑应作为独立包装层存在。

它负责：

- 创建标准 Bundle 目录
- 放置可执行文件
- 写入 `Info.plist`
- 放置 `.icns` 图标
- 输出到固定目录

它不负责：

- 改写聊天逻辑
- 改写网络逻辑
- 改写数据持久化逻辑

### 可重复生成

打包不应依赖一次性的 Finder 手工操作。

应通过一个固定脚本或固定命令稳定生成：

- 删除旧产物
- 重新构建
- 重新组装 `.app`

这样后续每次修改代码后，都可以稳定重新生成新的 `问.app`。

## 5. 技术方案

### 推荐方案

采用“Swift Package 主程序 + 独立 bundle 组装脚本”的方式。

### 原因

这种方式最符合当前项目状态：

- 改动最小
- 达成目标最快
- 风险最低
- 便于重复执行

### 产物结构

最终 `.app` 应符合标准 macOS 目录结构：

- `问.app/Contents/Info.plist`
- `问.app/Contents/MacOS/问`
- `问.app/Contents/Resources/AppIcon.icns`

其中：

- `Info.plist` 定义应用名、Bundle Identifier、图标绑定和可执行文件名
- `MacOS/问` 是实际可执行入口
- `Resources/AppIcon.icns` 是 Finder、Dock 和应用切换器使用的图标资源

## 6. 图标设计方案

### 视觉描述

图标采用安静、居中的单字布局：

- 画布为接近米白色的浅暖底色
- 使用圆角方形容器，贴近 macOS 图标观感
- `问` 字置中显示
- 文字颜色使用深棕黑或柔和深灰，保证在浅底上有足够对比

### 设计约束

- 字体观感需要稳定，避免细得过头导致小尺寸下发虚
- 背景不做高对比色块
- 阴影和渐变若存在，也只能非常弱，不能破坏“简洁”要求

### 导出要求

图标应生成 macOS 所需多尺寸源图，并最终合成为 `.icns`。

至少应覆盖从小尺寸 Finder 列表视图到 Dock 图标显示所需的常用尺寸。

## 7. 运行与打包流程

建议最终形成如下流程：

1. 构建应用可执行文件
2. 生成或复制图标资源
3. 创建 `dist/问.app/Contents/...` 目录
4. 写入 `Info.plist`
5. 复制主可执行文件到 `Contents/MacOS/问`
6. 复制 `AppIcon.icns` 到 `Contents/Resources/`
7. 必要时调整可执行权限
8. 产出最终 `dist/问.app`

## 8. 文件规划

建议新增以下类型文件：

### 打包文件

- 一个打包脚本，用于稳定生成 `.app`
- 一个 `Info.plist` 模板或程序化生成逻辑

### 图标文件

- 图标源文件
- 中间导出尺寸文件
- 最终 `.icns`

### 输出目录

- `dist/`

## 9. 错误处理与边界

### 需要明确处理的情况

1. 构建失败时，应停止打包而不是生成半成品
2. 图标生成失败时，应明确提示，而不是静默落回默认图标
3. 如果旧的 `dist/问.app` 已存在，应先清理后重新生成
4. 打包完成后应检查关键文件是否存在

### 当前已知限制

由于本次目标是“本机可双击打开”，不是“正式分发”，因此首次打开时仍可能受到 macOS 本地安全提示影响，具体表现取决于系统当前策略。但只要 bundle 结构正确、程序内容完整，它仍然应当作为本机应用正常启动。

## 10. 验证标准

本次实现完成后，必须满足：

1. 项目中成功生成 `dist/问.app`
2. `问.app` 图标显示为自定义“问”图标
3. Finder 中可以双击启动
4. 启动后能进入当前聊天窗口
5. 重新打包时可以稳定覆盖更新
6. 打包过程不依赖手工修改 bundle 内部文件

## 11. 最终建议

建议保持当前 Swift Package 主程序不变，只新增一层轻量 `.app` 包装与图标输出能力。

这样最适合当前阶段：

- 足够快
- 足够稳
- 足够接近真实 Mac 软件使用方式

在这一层完成后，后续如果要继续做：

- 正式签名
- 公证
- DMG
- 发布版工程整理

也都可以在现有基础上继续迭代，而不是推倒重来。

# Claude Lite Mac App 设计方案

## 1. 项目概述

目标是开发一个轻量的 macOS 桌面应用，通过 `api.tu-zi.com` 网关连接 Claude，并提供一个单一、连续的聊天窗口。

应用整体风格应当简洁、克制、易用，支持以下核心能力：

- 文本对话
- 文件传输
- 图像传输
- 连接状态展示
- 启动时自动检测连接
- 用户主动检测连接
- 本地保存聊天记录
- 仅限 Claude 不同版本模型切换

这是一个单会话应用，不是多会话工作台。

## 2. 已确认的产品方向

### 核心方向

- 只有一个主窗口
- 只有一条连续聊天线程
- 不做会话侧边栏
- 不做多标签布局
- 不在主界面堆复杂设置项
- 用尽量轻的方式展示状态与反馈

### 使用体验目标

这个 app 应更像一个原生的 Mac 聊天工具，而不是复杂的 AI 工作台。用户打开应用后，应当可以很自然地完成以下动作：

- 查看是否连通
- 选择 Claude 模型
- 输入消息
- 附加文件或图片
- 继续之前的聊天记录

## 3. 功能范围

### V1 范围内

1. 单一连续聊天窗口
2. 发送文本消息
3. 接收 Claude 回复
4. 发送前添加文件
5. 发送前添加图片
6. 顶部展示连接状态
7. 启动时自动检测连接
8. 用户手动检测连接
9. 安全保存 API key
10. 自动保存本地聊天记录
11. 仅允许切换 Claude 模型
12. 下次启动恢复上一次选择的 Claude 模型

### V1 暂不纳入

1. 多会话
2. 历史搜索
3. 提示词库
4. 云同步
5. 菜单栏模式
6. 高级参数配置面板
7. 团队协作能力

## 4. 界面布局

界面分为上下三块。

### 顶部状态栏

包含：

- 应用标题
- 当前连接状态
- 当前选中的 Claude 模型
- 手动检测连接按钮
- 轻量模型切换控件

### 中间聊天区域

包含：

- 用户消息
- Claude 回复
- 文件附件卡片
- 图片附件卡片
- 轻量发送中状态

这个区域是应用的主视觉焦点，应尽量干净。

### 底部输入区域

包含：

- 多行输入框
- 发送动作
- 添加文件动作
- 添加图片动作
- 待发送附件预览

输入区要保持简单，不应变成复杂工具栏。

## 5. 连接状态设计

应用同时支持安静的自动检测和用户主动触发检测。

### 连接状态定义

1. `Checking`
   正在检测服务器或凭证状态。

2. `Connected`
   服务可达，凭证有效。

3. `Disconnected`
   无法访问服务器，或服务器超时。

4. `Auth Failed`
   凭证缺失、无效，或被拒绝。

### 行为规则

- 应用启动后自动检测一次
- 检测成功时，用平静的绿色状态提示
- 检测失败时，在顶部显示简短错误状态
- 用户可以随时点击按钮再次检测
- 连接检测不能清空或破坏现有聊天记录

### 推荐检测接口

建议使用：

- `GET /v1/models`

原因：

- 请求轻量
- 适合启动时快速校验
- 不会发送真实聊天内容
- 已经完成过实际联通验证

## 6. 聊天行为设计

### 消息流转

1. 用户输入文本
2. 用户可选择添加文件或图片
3. 用户点击发送
4. 应用基于当前 Claude 模型和当前聊天上下文组装请求
5. Claude 回复后追加到同一条消息流
6. 本地自动保存会话内容

### 线程模型

- 只有一条持久化线程
- 之前的消息会继续作为上下文
- 重新启动应用后，应恢复同一条连续聊天

## 7. 模型切换设计

应用支持模型切换，但仅限 Claude 家族模型。

### 规则

- 模型选择器只显示 Claude 模型
- 网关中的非 Claude 模型一律隐藏
- 本地记住上次选择的 Claude 模型
- 若上次模型已经不可用，则自动回退到安全默认 Claude 模型

### 推荐行为

- 从 `GET /v1/models` 拉取模型列表
- 过滤出 Claude 相关模型
- 在顶部状态栏放置一个小型下拉选择器
- 控件保持轻量且始终可见

### 默认模型策略

- 默认使用一个稳定可用的 Claude 模型
- 顶部始终显示当前模型名
- 模型切换仅影响之后的新请求，不修改已有聊天记录

## 8. 文件与图片传输策略

即使后端附件传输细节后续有变化，界面层也应该先保持稳定、简单。

### 用户侧行为

- 用户可以通过系统文件选择器选中文件
- 用户可以通过系统文件选择器选中图片
- 已选中的内容在发送前显示为待发送附件卡片
- 发送后，附件卡片保留在聊天记录中

### 设计原则

界面层不应与某一种后端附件格式强绑定。

因此建议：

- 在应用内部定义统一的附件层
- 实际请求格式由传输层适配
- 如果后端要求先上传再引用，附件层负责处理
- 如果后端支持图片直接内联发送，附件层也可以兼容

### 当前确认程度

- 文本聊天路径已验证可用
- 模型列表路径已验证可用
- 文件与图片界面可以先确定
- 文件与图片最终请求字段，需要在开发阶段继续结合真实接口行为确认

## 9. 本地存储设计

### 需要本地保存的内容

- 聊天消息
- 附件元数据
- 上次选择的 Claude 模型
- 上次连接状态
- 保证连续使用所需的基础偏好设置

### 安全策略

- API key 存入 macOS Keychain
- 不把原始 key 以明文保存到普通配置文件
- 聊天记录保存到应用本地数据目录

### 持久化规则

- 每次成功完成一轮消息交互后自动保存
- 下次启动时自动恢复
- 后续可增加“清空本地记录”能力

## 10. API 接入说明

### 已确认可用的接口

1. `GET https://api.tu-zi.com/v1/models`
   已验证服务可达，并可在携带有效 Bearer Token 时返回模型列表。

2. `POST https://api.tu-zi.com/v1/messages`
   已使用 `claude-opus-4-6` 完成实际请求验证。

### 已完成的真实测试

已经发出过一条实际消息，请求模型只回复 `ok`，接口成功返回 Claude 响应，内容为 `ok`。

这说明以下关键链路已经跑通：

- 基础地址可访问
- Bearer 鉴权可用
- Claude 文本对话可用
- 指定 Claude 模型可通过该网关正常返回内容

## 11. 推荐技术方向

### 推荐结论

优先使用原生 macOS 技术路线。

### 建议的实现方向

- 使用 SwiftUI 构建主界面
- 使用 URLSession 发起网络请求
- 使用 Keychain 保存密钥
- 使用本地持久化保存聊天状态和用户偏好
- 使用独立的连接检测服务处理健康检查
- 使用独立的附件层处理文件和图片

### 为什么适合

- 最符合“轻量”目标
- 最容易做出干净的 Mac 原生体验
- 文件选择和图片选择集成自然
- 适合长期维护本地状态、连接状态和聊天体验

## 12. 内部模块划分

建议将应用划分为几个清晰模块。

### 模块建议

1. `ConnectionService`
   负责启动检测和手动检测

2. `ChatService`
   负责发送 Claude 聊天请求和接收回复

3. `ModelService`
   负责加载模型列表并过滤出 Claude 模型

4. `AttachmentService`
   负责标准化处理文件和图片输入

5. `PersistenceService`
   负责保存与恢复聊天记录、模型选择和本地状态

6. `SecureStore`
   负责从 Keychain 读写 API key

7. `ChatViewModel`
   负责协调单窗口聊天界面的状态

## 13. 启动流程

应用启动时建议执行：

1. 从 Keychain 读取保存的 API key
2. 读取本地聊天记录
3. 读取上次选择的 Claude 模型
4. 拉取模型列表
5. 过滤为 Claude 模型
6. 校验当前选中模型是否仍可用
7. 执行连接检测
8. 渲染恢复后的聊天界面

若没有有效 key：

- 显示一个很简单的凭证输入流程或内联提示
- 不应伪装成已连接状态

## 14. 错误处理

应用在出错时应该保持克制和可读。

### 原则

- 提示语尽量短
- 主界面不显示技术细节堆栈
- 保留旧聊天记录
- 让用户容易重试

### 建议的用户提示文案

- `Can’t reach server.`
- `Key not accepted.`
- `Selected model is unavailable.`
- `This file could not be attached.`
- `Image upload failed. Try again.`

## 15. V1 开发顺序

### 阶段 1

- 创建原生 app 基础壳子
- 完成单窗口界面
- 接入文本聊天

### 阶段 2

- 接入启动自动检测
- 接入手动检测
- 完成连接状态展示

### 阶段 3

- 接入本地聊天持久化
- 接入 API key 安全存储
- 接入模型加载与 Claude-only 模型切换

### 阶段 4

- 接入图片附件流程
- 接入文件附件流程
- 在聊天记录中展示附件卡片

### 阶段 5

- 处理边界情况
- 打磨错误提示
- 打磨恢复流程

## 16. 完成标准

当以下条件全部满足时，可以认为这一版完成：

1. 应用打开后进入一个干净的单窗口聊天界面
2. 应用启动时会自动检测连接
3. 用户可以主动检测连接
4. 用户可以成功与 Claude 对话
5. 用户只能在 Claude 模型之间切换
6. 选中的 Claude 模型会在重启后保留
7. 用户可以在窗口中添加文件和图片
8. 重新打开应用后能恢复聊天记录
9. API key 被安全存储
10. 错误状态可读且不会破坏已有数据

## 17. 当前剩余风险

目前最大的未确认项，不是整体产品形态，而是该网关在文件和图片传输上的最终实时请求格式细节。

这不会阻塞设计。

它只意味着在真正开发时，附件能力应当采用“可适配”的实现方式，而不是过早把请求格式写死。

## 18. 最终建议

建议按原生单窗口 macOS 应用推进。

整体原则是：

- 产品保持小而清晰
- 界面保持安静
- 状态提示保持克制

核心结构保持为：

- 一条连续线程
- 一个干净主窗口
- 一个可见但不喧宾夺主的状态区
- 一个轻量的 Claude-only 模型切换器
- 一套统一的文件与图片附件流程

这是在简单、可用和后续可维护之间最平衡的方案。
